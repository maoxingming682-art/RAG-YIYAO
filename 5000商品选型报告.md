## 二、实测基准数据

### 方案A：Prophet（原50商品demo方案）

| 指标 | 50商品demo | 5000商品实测 |
|------|-----------|-------------|
| 单商品训练 | 0.04-0.62秒 | 同（算法不随规模变化） |
| MAPE | 11-15% | 11%（1个成功样本） |
| 200商品成功率 | 100% | **0.5%（1/200）** |
| 5000串行外推 | - | ~2500秒（42分钟） |
| 5000并行8进程 | - | ~312秒（5.2分钟） |

**致命问题：** Prophet 1.3.0 + cmdstanpy 1.3.0 在Windows上有顽固bug——第一个商品训练后，stan_backend全局状态被破坏，后续全部失败。200个商品只有1个成功。英文路径也无效，是库本身的问题。

**结论：Prophet在Windows生产环境不可用，除非换Linux或修复cmdstanpy。**

### 方案B：ExponentialSmoothing（statsmodels）⭐ 实测冠军

| 指标 | 200商品实测 |
|------|------------|
| 单商品训练 | **0.054秒** |
| 200商品总耗时 | **12.7秒** |
| 吞吐率 | **15.7个/秒** |
| 成功率 | **100%（200/200）** |
| 平均MAPE | **14.1%** |
| 优秀(<15%) | 143个（71.5%） |
| 良好(15-25%) | 43个（21.5%） |
| 一般(25-35%) | 6个（3%） |
| 需改进(>35%) | 8个（4%） |
| 5000串行外推 | **269秒（4.5分钟）** |
| 5000并行8进程 | **34秒（0.6分钟）** |

**优势：** Windows原生稳定、statsmodels无外部依赖、100%成功率、速度比Prophet快3倍、精度MAPE 14.1%接近Prophet的11%。

### 方案C：Transformer时序模型（PatchTST/Autoformer，理论分析）

| 指标 | 理论值 |
|------|--------|
| 训练方式 | 多变量联合训练（一次4500个） |
| 精度 | 比Prophet高10-30%（MAPE预计9-12%） |
| 训练时间 | 一次训练30-60分钟（GPU） |
| 数据要求 | 至少2-3年历史数据 |
| 硬件要求 | GPU（4070Ti可跑） |
| 能捕捉品类关联 | ✅（感冒药+退烧药联动） |
| 能用多变量 | ✅（气温/促销/节假日） |

**劣势：** 365天数据量偏少（建议730+）、需GPU、调参复杂、可解释性弱、对5000商品的长尾商品不一定比简单模型好。

## 四、最优方案推荐

### 🏆 推荐方案：分层混合架构

```
┌──────────────────────────────────────────────────────┐
│ 5000商品分层预测方案                                    │
├──────────────────────────────────────────────────────┤
│                                                      │
│ A类头部商品（TOP 500，占销量70%）                      │
│   → ExponentialSmoothing 精调                        │
│   → 销量大、季节性强，值得单独建模                      │
│   → 500个 × 0.054秒 = 27秒                           │
│                                                      │
│ B类长尾商品（剩余4500个，占销量30%）                    │
│   → ExponentialSmoothing 批量跑                      │
│   → 4500个 × 0.054秒 = 243秒（4分钟）                 │
│   → 或用简单移动平均/季节性朴素（更快但精度略低）         │
│                                                      │
│ 全局监控层                                            │
│   → 数据漂移检测覆盖5000个                             │
│   → 漂移的单独触发重训                                 │
│   → LLM把数字转成补货建议+操作清单                     │
└──────────────────────────────────────────────────────┘
```

### 为什么选ExponentialSmoothing而不是Prophet？

1. **稳定性碾压**：200/200成功 vs 1/200成功。生产环境稳定性是第一位的。
2. **速度3倍**：0.054秒/个 vs 0.04-0.62秒/个，5000商品4.5分钟 vs 42分钟。
3. **精度接近**：MAPE 14.1% vs 11%，差距3个百分点，对库存补货决策足够。
4. **零外部依赖**：statsmodels是纯Python，不依赖cmdstanpy/C++编译，任何环境能跑。
5. **Prophet的核心优势（节假日建模+趋势分解）在5000商品场景下性价比低**——头部500个值得，长尾4500个不值得。

### 为什么不全部用Transformer？

1. **数据量不够**：只有365天历史，Transformer建议730+天。
2. **长尾商品不值得**：4500个长尾商品销量小规律弱，Transformer不一定比ExponentialSmoothing好。
3. **GPU资源+调参成本高**：为3%的精度提升付出10倍的工程成本不划算。
4. **可解释性**：店长需要理解"为什么建议备300盒"，ExponentialSmoothing能分解趋势+季节，Transformer是黑盒。

### 什么时候升级到Transformer？

当满足以下条件时值得升级：
- ✅ 历史数据积累到2年以上（730+天）
- ✅ 有气温/促销/竞品价格等外部变量
- ✅ 需要捕捉品类关联（如感冒药全家桶联动）
- ✅ 有GPU资源和算法工程师
- ✅ 精度提升3%能带来显著业务价值（千万级库存）
